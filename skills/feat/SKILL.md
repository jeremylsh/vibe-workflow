## Feat 开发助手

本 skill 是 vibe coding 工作流的起点，覆盖从需求到提交的完整开发循环。

### 工作流程

#### 第一阶段：理解需求

接收用户描述的需求，若信息不足则主动追问：
- 功能目标和用户场景
- 边界条件和不需要覆盖的范围
- 是否有 UI/API 接口变更

#### 第二阶段：Plan 模式（设计评审）

进入 plan 模式，输出设计方案：

```
## 需求理解
[用一两句话复述需求，确认理解正确]

## 实现方案
[主要实现思路及候选方案的取舍]

## 改动范围
[将要新建或修改的文件/模块]

## 关键决策
[有取舍的设计决策和原因]

## 不做的事
[明确排除在本次范围外的内容]
```

输出后**等待用户确认**，不要自动开始实现。用户反馈后按需调整，直到用户明确表示确认。

#### 第三阶段：创建 Feature 分支

用户确认后，退出 plan 模式，创建并切换到 feature 分支：

```bash
git fetch origin
git checkout master
git pull origin master
git checkout -b feat/<branch-name>
```

分支命名规则（英文、kebab-case、简短）：
- 新功能：`feat/<描述>`，例如 `feat/user-login`
- Bug 修复：`fix/<描述>`，例如 `fix/payment-timeout`
- 重构：`refactor/<描述>`

#### 第四阶段：实现

按照确认的方案实现代码：
- 每完成一个逻辑单元，简短报告进度
- 遇到方案偏差，先说明原因再调整
- 不做超出方案范围的改动

#### 第五阶段：触发提交

实现完成后自动触发 `/git-commit`，进入提交流程（提交后会自动发起 `/git-pr`）。

### 注意事项

- plan 阶段不写代码，等用户确认后再实现
- 始终从最新 master 拉取，避免冲突
- 分支名使用英文，简短清晰，不超过 4 个单词
- 实现完成后不要手动 push，交给 `/git-commit` → `/git-pr` 流程处理

### 示例触发语

- "帮我实现用户登录功能"
- "我需要一个导出 CSV 的接口"
- "修复支付超时的 bug"
- "/feat 商品详情页增加收藏按钮"

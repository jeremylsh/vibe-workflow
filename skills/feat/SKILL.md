## Feat 开发助手

本 skill 是 vibe coding 工作流的起点，覆盖从需求到提交的完整开发循环。支持两种模式：

- **Standard 模式**：每个关键节点（提交、PR、部署）均等待用户确认，适合需要精细控制的场景。
- **Fast 模式**：方案确认后全程自动执行，由对抗审查代替人工 review，仅最终部署前停下确认版本号和变更摘要。

---

### 模式选择

启动时**优先检查参数**：

- `/feat --fast <需求>` 或 `/feat <需求> --fast` → 直接进入 **Fast 模式**
- `/feat <需求>` 且未携带 `--fast` → 在需求澄清前询问一次：

```
Standard 还是 Fast 模式？

Standard：提交、PR、部署每步等你确认。
Fast：方案确认后全自动，仅最后部署前展示摘要和版本号确认一次。
```

模式选定后**不再询问**，整个流程严格按所选模式执行。

---

### 第一阶段：理解需求（两种模式相同）

接收用户描述的需求，若信息不足则主动追问：
- 功能目标和用户场景
- 边界条件和不需要覆盖的范围
- 是否有 UI/API 接口变更

---

### 第二阶段：Plan 模式（两种模式相同）

进入 plan 模式，输出设计方案：

```
## 需求理解
[用一两句话复述需求，确认理解正确]

## 实现方案
[主要实现思路及候选方案的取舍]

## 改动范围
[将要新建或修改的文件/模块]

## 关键决策
[有取舍的设计决策和原因]

## 不做的事
[明确排除在本次范围外的内容]
```

输出后**等待用户确认**，不要自动开始实现。用户反馈后按需调整，直到用户明确表示确认。

---

### 第三阶段：创建 Feature 分支（两种模式相同）

用户确认后，退出 plan 模式，创建并切换到 feature 分支：

```bash
git fetch origin
git checkout master
git pull origin master
git checkout -b feat/<branch-name>
```

分支命名规则（英文、kebab-case、简短）：
- 新功能：`feat/<描述>`，例如 `feat/user-login`
- Bug 修复：`fix/<描述>`，例如 `fix/payment-timeout`
- 重构：`refactor/<描述>`

---

### 第四阶段：实现（两种模式相同）

按照确认的方案实现代码：
- 每完成一个逻辑单元，简短报告进度
- 遇到方案偏差，先说明原因再调整
- 不做超出方案范围的改动

---

### 第五阶段：提交与合并

#### Standard 模式

实现完成后自动触发 `/git-commit`，进入提交流程（提交后会自动发起 `/git-pr`）。

---

#### Fast 模式 — 对抗审查 + 全自动执行

实现完成后，进入自动化流程，**无需用户干预**直到最终确认步骤。

##### 步骤 1：对抗代码审查（最多 2 轮修复）

切换为**批判性审查视角**，对照已确认方案逐项检查实现。

审查清单：
- 是否完整覆盖方案中所有改动点
- 有无明显 bug（边界条件、空指针、竞态）
- 有无安全漏洞（注入、越权、敏感信息泄露）
- 有无超出方案范围的多余改动
- 代码风格与现有代码库是否一致

**审查循环规则**：
- 发现阻塞问题 → 切回实现视角修复 → 重新审查，计为第 1 轮修复
- 修复后仍有阻塞问题 → 再次修复 → 重新审查，计为第 2 轮修复
- **第 2 轮修复后仍有阻塞问题 → 停止自动流程，输出人工仲裁报告**（见下方）

**通过**：输出审查结论并自动进入下一步（无需用户回复）：

```
[代码审查] 通过（第 N 轮）— 覆盖所有改动点，无安全隐患，无超出范围改动。
```

**人工仲裁报告**（超过修复上限时）：

```
[代码审查] 自动修复已达上限（2 轮），以下问题仍未解决，需要你来决策：

问题 1：[描述]
  审查方观点：[为何认为这是问题]
  实现方观点：[为何这样做]

问题 2：...

选项：
  A. 我来手动修复，修复后告诉我继续
  B. 忽略上述问题，强制继续自动流程
  C. 放弃本次 Fast 模式，切换回 Standard 模式逐步确认
```

等待用户选择后继续。

---

##### 步骤 2：自动提交

按 conventional commit 规范自动生成 message 并提交，无需用户确认：

```bash
git add <具体文件列表，不用 git add .>
git commit -m "<type>(<scope>): <subject>"
git push origin <branch> --set-upstream
```

##### 步骤 3：自动创建 PR

```bash
gh pr create \
  --base master \
  --title "<commit message 的 subject>" \
  --body "<一到两句话概括改动目的>" \
  --no-maintainer-edit
```

##### 步骤 4：对抗 PR 审查（最多 2 轮修复）

对 diff 做最终审查，确认与方案一致、无遗漏：

```bash
git diff origin/master...HEAD --stat
git diff origin/master...HEAD
```

审查循环规则与步骤 1 相同（最多 2 轮修复，超限则输出人工仲裁报告）。

**通过**：自动 approve 并合并：

```bash
gh pr review --approve -b "LGTM — fast mode auto review"
gh pr merge --merge --delete-branch
```

---

### 第六阶段：版本确认与部署（Fast 模式专属）

PR 合并后，输出本次变更摘要和建议版本号，**等待用户确认**，这是 Fast 模式唯一必须用户输入的步骤：

```
## Fast 模式完成，准备发布

**PR**: #<number> <URL>（已合并，分支已删除）
**代码审查**: 通过（共 N 轮）

### 本次变更摘要
**改动文件**（X 新增 / Y 修改）：
  src/auth/login.ts    （新增，+38）  JWT 登录接口，签发 access/refresh token
  src/routes/index.ts  （修改，+3）   注册登录路由

**Commit**: <hash> <message>

### 版本 Tag
建议版本：**v1.2.0**（基于上一版本 v1.1.3，本次为新功能 → minor 递增）

确认版本号后将自动打 tag 并部署。
直接回复"确认"使用建议版本，或回复你想要的版本号（如"v1.3.0"）。
```

用户确认版本号后，自动执行以下步骤，**无需额外输入**：

1. **打版本 tag**：创建带注释 tag 并推送到远端。
2. **部署**：拉取最新 master，执行 `bash deploy.sh`，实时输出日志。

部署成功后输出结果；失败则停止并报告错误，等待人工处理。

---

### 注意事项

- plan 阶段不写代码，等用户确认后再实现
- 始终从最新 master 拉取，避免冲突
- 分支名使用英文，简短清晰，不超过 4 个单词
- Standard 模式：实现完成后不要手动 push，交给 `/git-commit` → `/git-pr` 流程处理
- Fast 模式：对抗审查每轮修复后必须重新完整过审查清单，不得跳项
- Fast 模式：修复轮次计数在代码审查和 PR 审查中各自独立计算（各最多 2 轮）

---

### 示例触发语

- "帮我实现用户登录功能"（进入模式选择）
- "/feat --fast 商品详情页增加收藏按钮"（直接进入 Fast 模式）
- "我需要一个导出 CSV 的接口"
- "修复支付超时的 bug"
- "/feat 用户登录 --fast"
